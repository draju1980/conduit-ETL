Pull data from heterogeneous sources, transform with SQL, validate with thirty built-in expectations, and load to ten destinations — all from a single CLI tool built on Deno + TypeScript.

deno install -Agf -n conduit jsr:@conduit/etl/cli

01Overview

Conduit-ETL is an open-source ELT workbench that runs on your laptop, on a server, or anywhere Deno runs. Pipelines are described in a single pipeline.yaml file and executed by a single binary — no orchestrator cluster, no cloud account required, no configuration sprawl.

The product is built around four convictions: pipelines should be declarative, transforms should be SQL, data quality should be first-class, and the whole thing should be local-first. You can run a complete extract → transform → validate → load cycle without leaving your terminal.

Status — Core pipeline (extract / normalize / transform / validate / load), the CLI, the connector framework, and 30 GE-style validators are shipped and tested (81 passing tests). The daemon, scheduler, web UI, vault, and time-machine features are scoped in PLAN.md and still in progress.

02Features

01 · ENGINE

SQL transforms via DuckDB

JOINs, CTEs, window functions, and aggregations across every source — uniform SQL semantics regardless of where the data came from.

02 · LOAD

Ten destination types

CSV, JSON, JSONL, Parquet, PostgreSQL, MySQL, Snowflake, BigQuery, MongoDB, and S3 — all driven by the same loader interface.

03 · VALIDATE

30 GE-style expectations

Table, column, aggregate, set membership, regex, uniqueness, numeric range, and string-length checks with a mostly partial-pass threshold.

04 · POLICY

Per-check failure policy

Each expectation can fail or warn. Validation reports land in .conduit/reports/ as JSON for downstream tooling.

05 · CONFIG

YAML-driven pipelines

One declarative file describes sources, transforms, validation, and destinations. Schema-validated by Zod at load time.

06 · CONNECTORS

Pluggable connector CLI

conduit source add / rm / enable / disable / list — and the same for destinations. Modules can be turned on per-project.

07 · DISTRIBUTION

JSR-distributed

Install with one Deno command on macOS, Linux, or Windows. Native dependencies (DuckDB, drivers) resolve automatically per platform.

08 · LIBRARY

Use as a TypeScript library

Import runPipeline, loadPipeline, or any individual stage — embed Conduit inside your own Deno application.

03Architecture

Every pipeline runs through five stages. The normalize step is the hinge of the system — it converts whatever shape a connector produces into a uniform DuckDB table so the SQL engine never has to care where the data came from.

The normalize step

Every connector produces a DataTable — plain JS objects with column metadata. The normalize step writes each table to a temp CSV, uses DuckDB's read_csv with auto-detect to infer types, and registers it as an in-memory DuckDB table. From that point on, transform and validation see a single, consistent SQL surface.

When new source connectors are added later (PostgreSQL, MongoDB, etc.), they only need to produce a DataTable. The normalize step does the rest.

04Install

Conduit is distributed via JSR. One install command works on macOS (Intel + Apple Silicon), Linux, and Windows.

Prerequisite

curl -fsSL https://deno.land/install.sh | sh

Install the CLI globally

deno install -Agf -n conduit jsr:@conduit/etl/cli

Flags: -A all permissions (required for DuckDB + database drivers), -g install globally, -f force overwrite, -n conduit name the binary conduit.

Verify

conduit version

Update / uninstall

# pull latest from JSR
deno install -Agf -n conduit jsr:@conduit/etl/cli

# remove
deno uninstall conduit

05Quick start

  1. Initialize a project

    Scaffold a sample pipeline.yaml and the .conduit/ working directory.

    conduit init

  2. Edit your pipeline

    Open pipeline.yaml and define your sources, transform SQL, validation checks, and destinations. (See the full example below.)

  3. Run it

    # execute the full pipeline
conduit run pipeline.yaml

# validate without loading
conduit run pipeline.yaml --dry-run

# run validation suite only
conduit validate pipeline.yaml

  4. Read the report

    Validation results land in .conduit/reports/ as timestamped JSON. Failures with on_failure: fail abort the load; warn logs and continues.

Use as a library

import { runPipeline, loadPipeline } from "jsr:@conduit/etl";

await runPipeline("pipeline.yaml");

06Pipeline YAML

A pipeline is one declarative file. Here's a complete working example with two sources, a transform, mixed legacy + GE-style validation, and a CSV destination:

pipeline:
  name: orders_enriched

sources:
  - name: orders
    type: csv
    config:
      path: data/orders.csv

  - name: regions
    type: csv
    config:
      path: data/regions.csv

transform:
  sql: |
    SELECT o.order_id, o.customer_id, o.amount, o.status, r.region
    FROM orders o
    JOIN regions r ON o.region_id = r.id
    WHERE o.status = 'active'

validation:
  # Legacy schema check
  - type: schema
    columns:
      - { name: order_id, type: INTEGER }
      - { name: amount,   type: DECIMAL }
    on_failure: fail

  # GE-style: at least 95% of statuses must be in this set
  - expectation_type: expect_column_values_to_be_in_set
    kwargs:
      column: status
      value_set: [active, completed, cancelled]
      mostly: 0.95
    on_failure: fail

  # GE-style: row count sanity
  - expectation_type: expect_table_row_count_to_be_between
    kwargs:
      min_value: 1
      max_value: 10000000
    on_failure: warn

destinations:
  - name: output
    type: csv
    mode: full_refresh
    config:
      path: output/result.csv
Tip — Both legacy type: checks and GE-style expectation_type: checks can be mixed in the same validation: block. Each check independently sets on_failure: fail or warn.

08Validation catalog

Conduit ships with 30 GE-style expectations plus 4 legacy check types. All run on the transformed result via DuckDB SQL. Column-map expectations support a mostly threshold (0.0 – 1.0) for partial-pass behavior.

Legacy checks

·4 original format
  • schemaVerify output columns match expected names and types
  • null_checkAssert specified columns contain no nulls
  • row_countCatch empty result sets or unexpected row counts
  • customArbitrary SQL assertion — rows returned mean failures

Table-level

·6
  • expect_table_row_count_to_equalRow count equals a specific value
  • expect_table_row_count_to_be_betweenRow count within a min/max range
  • expect_table_column_count_to_equalNumber of columns equals a specific value
  • expect_table_column_count_to_be_betweenNumber of columns within a range
  • expect_table_columns_to_match_ordered_listColumns match an exact ordered list
  • expect_table_columns_to_match_setColumns match a set, order-independent

Column existence & type

·3
  • expect_column_to_existColumn exists in the table
  • expect_column_values_to_be_of_typeColumn has a specific DuckDB type
  • expect_column_values_to_be_in_type_listColumn type is one of a list

Completeness

·2 supports mostly
  • expect_column_values_to_not_be_nullNo null values, or within a mostly threshold
  • expect_column_values_to_be_nullAll values are null

Set membership

·4 supports mostly
  • expect_column_values_to_be_in_setEach value is in a given set
  • expect_column_values_to_not_be_in_setNo value is in a given set
  • expect_column_distinct_values_to_equal_setDistinct values exactly match a set
  • expect_column_distinct_values_to_contain_setDistinct values contain a required set

Numeric

·3 supports mostly
  • expect_column_values_to_be_betweenEach value within min/max (optional strict_min / strict_max)
  • expect_column_values_to_be_increasingValues are monotonically increasing
  • expect_column_values_to_be_decreasingValues are monotonically decreasing

Uniqueness

·1 supports mostly
  • expect_column_values_to_be_uniqueAll values are unique — no duplicates

String pattern & length

·4 supports mostly
  • expect_column_values_to_match_regexEach value matches a regex pattern
  • expect_column_values_to_not_match_regexNo value matches a regex pattern
  • expect_column_value_lengths_to_equalString length equals a specific value
  • expect_column_value_lengths_to_be_betweenString length within a range

Aggregate statistics

·7
  • expect_column_min_to_be_betweenColumn minimum within a range
  • expect_column_max_to_be_betweenColumn maximum within a range
  • expect_column_mean_to_be_betweenColumn mean within a range
  • expect_column_median_to_be_betweenColumn median within a range
  • expect_column_stdev_to_be_betweenStandard deviation within a range
  • expect_column_sum_to_be_betweenColumn sum within a range
  • expect_column_unique_value_count_to_be_betweenDistinct value count within a range

The mostly parameter

Column-map expectations accept a mostly field between 0.0 and 1.0 — the proportion of non-null values that must pass. mostly: 0.95 means up to 5 % unexpected values are tolerated:

- expectation_type: expect_column_values_to_not_be_null
  kwargs:
    column: email
    mostly: 0.95
  on_failure: fail

Result format

Every expectation returns rich diagnostics in the JSON report: element_count, unexpected_count, unexpected_percent, missing_count / missing_percent, a partial_unexpected_list sample of up to 20 failing values, and an observed_value summary.

09Destinations

Ten loader implementations ship in-tree. All are driven by the same loader interface — register a connector, point at credentials, write to it.

csv
json
jsonl
parquet

mysql
snowflake

Configure each destination's mode (full_refresh / incremental), batch_size, optional merge keys for incremental loads, and connection details inline in pipeline.yaml or via the connection: field referencing the secret vault.

The .conduit/ directory

Every project has a .conduit/ working directory created by conduit init. It holds:

  • checkpoints/ — pipeline checkpoint state
  • logs/ — execution logs
  • reports/ — JSON validation reports, one per run
  • scheduler/ — daemon state when conduit up is running
